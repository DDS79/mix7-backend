Ниже простая версия статуса проекта и плана на 2 недели для нового разработчика.

## Статус проекта сейчас

Сейчас проект уже **не на нуле** и не в состоянии “сырой идеи”.  
У нас уже есть рабочая основа.

Что уже есть:
- backend как главный источник правды
- web frontend с событиями, регистрацией, билетом и checkout handoff
- Telegram bot как второй клиент той же системы
- ticket flow
- accessCode
- базовая инженерная система контроля:
  - отчёты
  - task protocol
  - alerts

То есть проект уже можно назвать **рабочим phase-1 фундаментом**.

Но важно понимать:
он ещё не “спокойный production”.  
Сейчас это стадия, где система уже собрана, но ещё требует доведения, проверки и уплотнения.

Если совсем просто:
- скелет уже есть
- основные органы уже работают
- теперь нужно довести всё до состояния нормальной, устойчивой жизни

---

## Что сейчас главное

Главная задача сейчас — **не придумывать много нового**, а сделать уже собранное стабильным.

То есть фокус ближайших двух недель:
- меньше новых сущностей
- меньше новых направлений
- больше проверки и доведения того, что уже есть

---

# Roadmap на 2 недели

## Неделя 1 — стабилизация существующего phase-1

### 1. Проверить все главные пользовательские сценарии
Нужно пройти руками и технически:

- web:
  - список событий
  - event detail
  - free registration
  - paid registration
  - ticket page
  - checkout handoff

- Telegram:
  - список событий
  - регистрация на free event
  - paid handoff в web
  - просмотр билета / кода

Цель:
- убедиться, что цепочки работают не “в теории”, а как единый путь

---

### 2. Найти и убрать остаточные разрывы между слоями
Проверять:
- frontend ↔ backend
- telegram ↔ backend
- registration ↔ ticket
- registration ↔ checkout
- session ↔ user flow

Цель:
- убрать места, где система формально собрана, но ещё может разваливаться на стыках

---

### 3. Проверить ticket/use flow
Нужно посмотреть:
- понятно ли выглядит билет
- понятен ли `accessCode`
- не путаются ли ticketId / QR / код для охраны
- удобно ли этим реально пользоваться

Цель:
- чтобы билет был не просто “данные на экране”, а реально понятный operational artifact

---

### 4. Проверить paid flow особенно внимательно
Paid path всегда рискованнее.

Нужно:
- проверить handoff в checkout
- проверить, что order / orderId / payment start / confirm живут согласованно
- проверить, что нет скрытых контрактных дыр

Цель:
- чтобы платный путь был таким же понятным и честным, как бесплатный

---

## Неделя 2 — доведение Telegram и инженерной дисциплины

### 5. Довести Telegram как реальный канал
На второй неделе важно не просто “иметь бота”, а сделать его реально удобным каналом.

Нужно:
- довести UX Telegram
- убедиться, что события, регистрации и билеты показываются внятно
- проверить, что человек в Telegram не теряется между шагами
- проверить, что Telegram не расходится с web по продуктовой правде

Цель:
- Telegram должен быть полноценным phase-1 интерфейсом, а не просто дополнительной оболочкой

---

### 6. Закрепить task protocol в реальной работе
Уже есть система с:
- `MIX7-XXX`
- `start / work / done / blocked`
- reporting
- alert layer

Теперь это нужно реально начать использовать дисциплинированно.

Что делать:
- следить, чтобы новые commits шли по протоколу
- не накапливать “безымянную работу”
- смотреть, как отчёты и alerts начинают отражать реальное состояние

Цель:
- чтобы инженерная система начала работать не как эксперимент, а как нормальный operating layer

---

### 7. Подчистить repo hygiene
Нужно:
- уменьшить шум
- убрать лишние артефакты
- улучшить чистоту рабочего дерева
- следить, чтобы source of truth оставался понятным

Цель:
- чтобы в системе было меньше ложного инженерного шума
- чтобы отчёты и alerts были чище

---

# Что не нужно делать в эти 2 недели

Очень важно:  
**сейчас не надо резко расширять продукт**.

То есть пока не приоритет:
- новые большие модули
- сложные member flows
- админка
- check-in platform
- новые каналы
- большой редизайн

Почему:
потому что сейчас важнее сделать уже существующий phase-1 устойчивым.  
Иначе проект будет расти вширь быстрее, чем укрепляться внутри.

---

# Главный смысл ближайших 2 недель

Если сказать одной фразой:

**Нужно превратить уже собранный phase-1 из “рабочей конструкции” в “устойчивую рабочую систему”.**

---

# Что должен делать новый разработчик каждый день

Очень простое правило:

1. Брать один реальный кусок flow.
2. Проверять его end-to-end.
3. Искать точный разрыв, если он есть.
4. Чинить минимально.
5. Проверять после фикса.
6. Не плодить новую архитектуру без необходимости.

---

# Какой хороший результат через 2 недели

Через 2 недели должно быть так:

- web flow стабилен
- Telegram flow стабилен
- free и paid paths понятны и воспроизводимы
- билет и accessCode понятны пользователю
- checkout path не разваливается на стыках
- task/report/alert system реально используется
- repo чище
- команда лучше видит реальное состояние проекта

---

# Если совсем коротко для нового разработчика

Вот короткая версия, которую можно ему просто отправить:

- Сейчас проект уже собран как phase-1 основа.
- Главная задача ближайших 2 недель — не изобретать новое, а стабилизировать уже собранное.
- Неделя 1:
  - проверить и довести web flow
  - проверить и довести free / paid / ticket / checkout
- Неделя 2:
  - довести Telegram
  - закрепить task protocol
  - подчистить repo/process hygiene
- Основной принцип:
  - backend — источник правды
  - web и Telegram — тонкие клиенты
  - изменения должны быть минимальными, точными и проверяемыми
