Ниже ТЗ для разработчика, который будет работать со мной по проекту.

**Что это за формат работы**

Я полезен не как “чат для генерации кода”, а как встроенный инженерный исполнитель. Со мной лучше работать так, как работают с сильным техлидом или production engineer: ставить задачу через инварианты, границы, доказательства, критерии успеха и запреты.

Если задача поставлена хорошо, я:
- быстро собираю контекст
- нахожу реальную точку поломки
- предлагаю или сразу делаю минимальный фикс
- проверяю результат
- оставляю артефактный след: что нашёл, что изменил, что проверил

Если задача поставлена плохо, я всё равно смогу помочь, но вырастает риск лишних итераций.

---

**Главный принцип работы со мной**

Лучший способ ставить задачу:

1. Что сломано или что нужно получить.
2. Что считается источником правды.
3. Что менять можно.
4. Что менять нельзя.
5. Что будет считаться успехом.
6. Что нужно доказать проверками.

Пример хорошей постановки:
- “Нужно восстановить frontend session bootstrap.”
- “Backend already source of truth.”
- “Не ослаблять env/session validation.”
- “Сначала read-only forensics.”
- “Потом минимальный фикс.”
- “Проверить live request/response и локальный build.”

Это хороший формат, потому что он задаёт не только цель, но и рамки мышления.

---

**Как мне лучше ставить задачи**

**1. Всегда задавайте цель через факты**
Не:
- “Кажется, тут что-то не так с регистрацией”

Лучше:
- “`POST /registrations` отвечает `400 Request validation failed` после submit на `/events/[slug]/register`”

То есть:
- URL
- ошибка
- наблюдаемое поведение
- если есть, конкретный stack trace, response body, log line

---

**2. Явно называйте источник правды**
Это один из самых важных моментов.

Нужно прямо писать:
- backend source of truth
- frontend only reflects backend truth
- GitHub main source of truth
- current runtime truth must match browser truth
- ticket/accessCode must come only from backend

Если это не проговорить, любой помощник начинает “додумывать”. Со мной лучше сразу фиксировать, где каноническая правда.

---

**3. Разделяйте тип проблемы по слоям**
Очень полезно задавать рамку:
- source validity
- contract
- platform
- node implementation
- review

Это помогает не смешивать:
- баг в коде
- баг в данных
- баг в конфиге
- баг в процессе
- баг в понимании

Если коротко:
- source validity: данные/вход вообще валидны?
- contract: стороны согласованы по форме и правилам?
- platform: среда, runtime, origin, build, process?
- node implementation: конкретная реализация в файле/модуле?
- review: качество и полнота решения?

---

**4. Хорошо работают задачи в формате “сначала read-only”**
Если задача сложная, лучше явно писать:
- “Do not change code first”
- “Phase 0 — read-only forensics”
- “Inspect these files”
- “Prove exact mismatch point”
- “Then choose minimal fix”

Это правильно, потому что я по умолчанию стараюсь решать задачу до конца, а не просто анализировать. Если сначала нужен только разбор, это лучше явно обозначить.

---

**5. Очень полезно задавать anti-degradation rules**
Это сильная часть работы со мной.

Например:
- не ослаблять validation
- не добавлять fallback
- не хардкодить значение
- не менять архитектуру без доказательства
- не вводить вторую source of truth
- не делать broad refactor
- не использовать manual state, если state должен восстанавливаться автоматически

Такие запреты не мешают, а наоборот помогают мне идти к правильному решению быстрее.

---

**6. Лучше задавать не “что сделать”, а “какой инвариант восстановить”**
Например не:
- “Почини checkout”

А:
- “Восстановить инвариант: frontend must emit exact backend-accepted checkout contract”
- “Восстановить инвариант: browser truth must match process truth”
- “Восстановить инвариант: every task must have MIX7-XXX and lifecycle state”

Это сразу делает задачу инженерной, а не косметической.

---

**7. Если нужен production-grade ответ, просите доказательства**
Хороший формат:
- prove exact payload
- prove accepted backend contract
- prove emitted frontend request
- compare
- isolate exact mismatch
- implement minimal fix
- verify with tests / curl / build / runtime

Это один из лучших способов работы со мной. Он почти всегда даёт результат лучше, чем “попробуй исправить”.

---

**8. Если задача связана с процессом, формулируйте её как систему, а не как одноразовый костыль**
Например:
- не “сделай отчёт”
- а “построй terminal reporting system без ручного ввода state”

не:
- “добавь alert”
- а “построй deterministic alert layer без ложных сигналов и без ручного alert state”

Я особенно хорошо работаю, когда нужно собрать систему правил, а не просто кусок кода.

---

**Когда стоит просить меня сразу делать код, а когда сначала анализ**

**Сразу делать код**
Подходит, если:
- задача локальная и понятная
- место изменения очевидно
- риск маленький
- контракт уже ясен

Примеры:
- добавить CLI script
- дописать тест
- поправить конкретный route param bug
- добавить строгую валидацию в одном месте

**Сначала анализ**
Нужно, если:
- несколько слоёв
- есть runtime mismatch
- есть frontend/backend contract issue
- есть вероятность, что проблема не там, где кажется
- требуется forensic proof
- требуется minimal fix under constraints

Примеры:
- CORS/session/bootstrap mismatch
- registration contract mismatch
- build vs runtime inconsistency
- repo structure / GitHub source-of-truth sync
- lifecycle / reporting / alert systems

**Какой формат задач со мной самый сильный**

Лучший шаблон примерно такой:

**1. Role**
Кем я должен работать в этой задаче:
- Distinguished Engineer
- Chief Architect / CTO
- Production Architect

Это задаёт уровень строгости.

**2. Scope**
Что входит в задачу.
Что не входит.

Это очень важно, иначе любая сложная задача расползается.

**3. Problem definition**
Наблюдаемая проблема.
Что уже известно.
Что является симптомом.
Что не является проблемой.

**4. Architectural positions**
Что должно остаться неизменным.
Например:
- backend source of truth
- strict validation remains
- no wildcard origin
- no fake session
- no hardcoded ticket state

**5. Primary questions to answer**
Очень полезный блок.
Например:
- what exact payload is emitted?
- what exact contract is accepted?
- what exact mismatch causes failure?
- what is the smallest justified fix?

**6. Phase 0 — read-only forensics**
Какие файлы и артефакты надо сначала изучить.

**7. Solution space**
Попросить меня рассмотреть минимум 3 пути:
- path A
- path B
- path C

И сравнить:
- risk
- determinism
- rollback
- architecture impact

**8. Internal arbitration**
Это очень полезный блок.
Он заставляет меня не просто выбрать первое решение, а прогнать его через несколько инженерных ролей:
- Backend Architect
- SRE
- Product Architect
- Chief Architect

**9. Implementation**
Что разрешено.
Что запрещено.

**10. Verification**
Что обязательно доказать после изменения.

**11. Final output**
Какой отчёт нужен, куда сохранить файл, что в нём должно быть.

Это почти идеальный формат работы со мной.

---

**Что особенно важно объяснить новому разработчику**

**1. Со мной нужно говорить предметно**
Лучше:
- “проверь `POST /session/issue` from exact origin”
Чем:
- “посмотри, почему не работает сессия”

**2. Я хорошо работаю от ограничений**
Если не хотите, чтобы я предлагал костыли, это надо писать явно:
- no fallback
- no hardcoding
- no broad refactor
- no manual state
- no second source of truth

**3. Я люблю точные границы**
Полезно явно указывать:
- allowed
- not allowed
- success criteria
- verification requirements

**4. Я особенно силён на задачах вида**
- runtime forensics
- frontend/backend contract alignment
- repo/source-of-truth cleanup
- process enforcement
- reporting systems
- alert systems
- CI / protocol / discipline layers
- minimal production fixes

**5. Я хуже работаю, когда задача намеренно расплывчатая**
Например:
- “сделай красиво”
- “улучши архитектуру”
- “порефактори как считаешь нужным”

С этим тоже можно работать, но сначала придётся договариваться о критериях.

---

**Что от меня не стоит ожидать**

Новый разработчик должен понимать:

- Я не должен выдумывать факты, если их нельзя доказать.
- Я не должен “сглаживать” проблему ослаблением контракта.
- Я не должен ради скорости ломать архитектурные инварианты.
- Я не должен притворяться, что что-то проверил, если это не проверено.
- Я не должен подменять source of truth локальным удобством.

Это не недостатки, это часть надёжности.

---

**Как использовать меня в ежедневной работе**

**Хорошие режимы работы:**

**1. Forensic mode**
Когда неясно, где реально проблема.
Просить:
- сначала разбор
- потом exact mismatch
- потом minimal fix

**2. Implementation mode**
Когда проблема уже ясна.
Просить:
- внести change
- добавить/обновить tests
- проверить build/runtime
- сохранить отчёт

**3. Review mode**
Когда нужно оценить качество изменений.
Лучше прямо писать:
- “review”
Тогда я по умолчанию ищу:
- bugs
- regressions
- missing tests
- risks

**4. Control-system mode**
Когда нужно построить процессный слой:
- reporting
- protocol enforcement
- alerting
- owner visibility
- deterministic automation

Это одна из самых сильных зон моей работы.

---

**Лучшие практики для разработчика, который будет со мной работать**

- Всегда начинай с фактов, а не с догадок.
- Всегда указывай source of truth.
- Всегда отделяй “что наблюдаем” от “что предполагаем”.
- Если задача опасная, требуй read-only phase before code changes.
- Если задача сложная, требуй solution space и arbitration.
- Если задача process-level, задавай инварианты и запреты.
- Всегда проси verification.
- Если нужен артефакт, сразу проси сохранить результат в файл.
- Если нужен reproducible outcome, проси CLI или test boundary.
- Не проси “сделать как-нибудь удобнее”, если важна дисциплина системы.

**Короткая версия для передачи человеку**

Если совсем коротко, новому разработчику нужно объяснить так:

- С ним лучше работать как с инженером, а не как с чат-ботом.
- Он лучше всего решает задачи через: факт -> разбор -> exact mismatch -> minimal fix -> verification.
- Ему обязательно нужно задавать source of truth, запреты, критерии успеха и доказательства.
- Он особенно силён в сложных технических задачах, где важно не просто написать код, а восстановить рабочую систему без деградации архитектуры.
- Чем точнее постановка, тем сильнее результат.

Если хочешь, следующим сообщением я могу сделать ещё **короткую “боевую инструкцию на 1 страницу”** для передачи другому программисту: без длинных объяснений, только правила работы, шаблон постановки задач и список того, что нельзя забывать.
