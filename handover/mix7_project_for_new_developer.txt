**Как для нового разработчика**

Это проект про клубные события, регистрацию, билеты и проход, но его важно воспринимать не как “сайт + бот”, а как одну систему с единым backend.

Главная идея такая:  
backend хранит всю продуктовую правду, а web и Telegram только используют её.  
То есть если тебе нужно понять, “как тут всё работает”, всегда начинай с backend, а не с интерфейсов.

**Что делает система**
- отдаёт список событий
- показывает детали события
- регистрирует человека на событие
- для бесплатного события выдаёт билет
- для платного события отдаёт checkout handoff
- хранит и возвращает `accessCode`, ticket, order, session

**Какие здесь основные части**
- **Backend**  
  Главное ядро. Здесь контракты, валидация, события, регистрации, билеты, checkout handoff, session issuance.
- **Web frontend**  
  Сайт, который показывает события, регистрацию, билет, checkout.
- **Telegram bot**  
  Второй клиент той же системы. Он не живёт по своим правилам, а должен использовать тот же backend.
- **Engineering control layer**  
  Это уже внутренний слой для команды: task protocol, reporting, alerts, owner visibility.

**Как правильно понимать архитектуру**
- backend = source of truth
- frontend = thin client
- telegram = thin client
- ticket/accessCode/checkout нельзя придумывать в клиентах
- если есть спор между UI и backend, прав backend
- если что-то “удобно сделать на фронте”, но это ломает truth model, так делать нельзя

**Как лучше заходить в задачу**
Если тебе дают фичу или баг:
1. Сначала пойми, где источник правды.
2. Потом проверь контракт между слоями.
3. Потом ищи минимальное место исправления.
4. Только после этого меняй код.

То есть не нужно начинать с “я сейчас быстро поправлю страницу”.  
Очень часто проблема не в странице, а в:
- backend contract
- runtime/env
- session propagation
- route params
- origin policy
- handoff между шагами

**Какие типы задач здесь встречаются**
- contract alignment между frontend и backend
- runtime / env / origin / session проблемы
- registration / ticket / checkout flow
- Telegram integration как thin client
- process tooling: reports, task protocol, alerts

**Что особенно важно в этой кодовой базе**
- здесь нельзя лечить проблему fallback’ом, если ломается контракт
- нельзя ослаблять validation, чтобы “прошло”
- нельзя добавлять вторую source of truth
- нельзя хардкодить значения, которые должен выдавать backend
- нельзя путать локальную активность с реальным системным состоянием

**Если очень коротко**
Твоя главная задача как разработчика в этом проекте:
не “писать фичи в интерфейсе”, а аккуратно двигать одну систему, где backend управляет правдой, а остальные слои должны честно ей следовать.

Если хочешь, следующим сообщением я опишу проект во втором формате:
**как для инвестора/фаундера, но без пафоса**.
