3. Solution space

Path A — Manual `tasks.json`
- rejected
- violates invariant: no manual task state maintenance
- low reliability
- invites stale and unverifiable data

Path B — Git-driven reconstruction only
- strong deterministic base
- good for completed tasks
- insufficient alone for truthful `in progress`, because owner also needs current working state
- fails to capture dirty tree without commits

Path C — Hybrid (`git + working tree artifacts`)
- chosen
- completed tasks: from commit history only
- in-progress tasks: from current filtered working tree only
- blocked tasks: only if explicit git/sandbox/conflict signal exists
- deterministic, reproducible, and more truthful than pure commit-only reconstruction

Path D — External system
- rejected
- adds a second source of truth
- violates minimality

4. Internal AI arbitration

Backend Architect:
- requires deterministic source
- rejects manual state
- accepts git as primary truth

SRE:
- requires reproducibility
- rejects undocumented heuristics
- requires filtering of `.next`, env files, and generated junk

Product Architect:
- requires Russian output
- requires report readable in under 2 minutes
- requires owner-visible sections:
  - completed
  - in progress
  - blocked
  - risks
  - owner decisions

Chief Architect:
- chooses Path C
- reason:
  - pure git works for completed work only
  - owner-level reality also needs current task state
  - current task state is only truthfully available from dirty tree
  - therefore:
    - commits = completed truth
    - dirty tree = current work truth
    - no manual override layer

5. Chosen architecture

Source of truth:
- primary: git history
- secondary: current working tree
- forbidden: manual `tasks.json`

Pipeline:
- collect
  - `git log`
  - `git numstat`
  - `git status --short --branch -uall`
  - `git ls-files --others --exclude-standard`
- derive
  - completed tasks from commits on selected date
  - in-progress tasks from filtered dirty tree
  - blocked tasks only from explicit conflict/block signals
- score
  - quality `0–10`
  - effort band
  - risks and owner decisions
- build
  - Russian owner-readable text report
  - deterministic JSON state snapshot
- output
  - stdout
  - `project-control/reports/*.txt`
  - `project-control/state/*.json`

Grouping rules:
- first choice: explicit `MIX7-XXX` from commit subject
- fallback: normalized commit subject
- working tree groups: path-area mapping from config

Important invariant:
- there are no completed tasks without commits
- there are no blocked tasks without explicit proof
- there is no hidden task database
